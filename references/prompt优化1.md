## 第一部分：AI Hardness 工程专家组发现

### 🔴 发现1：隐式指令冲突——"自然整合" vs "显性标注张力"

**Instruction Following 专家：**

v8.0.0 中存在一个**自相矛盾的指令对**：

> 第67行："Integrate their perspectives naturally — do not mechanically give each expert a speaking turn."
>
> 第68行："**However:** when the three perspectives produce genuine tension on an issue, you MUST surface that tension explicitly and present it as teaching material."

这两条指令之间的边界是模糊的。LLM 需要在每个问题上做一个判断："这里有没有genuine tension？" 但 prompt 没有给出判断标准。实际结果是：

- **保守模式：** LLM 倾向于"自然整合"（因为这是默认指令），很少触发"显性标注张力"——导致输出变得单声道
- **激进模式：** LLM 为了满足"MUST surface tension"，在没有真实张力的地方制造人工张力——导致输出显得刻意

**建议：** 给出**张力触发条件**的具体定义，而不是让 LLM 自行判断：

```markdown
当以下任一条件成立时，显性标注专家张力：
- 经文的原始语境含义与直觉的职场应用之间存在落差
- 恩典原则与组织公平原则指向不同的行动方向
- 短期忠于信仰可能导致可预见的职场损失
其余情况下，自然整合即可。
```

### 🔴 发现2：Internal Pre-computation 是一个"幻觉指令"

**Attention 专家：**

v8.0.0 要求 LLM 在生成输出前"silently perform"四步预计算（Genre Scan → Core Tension → Risk Assessment → Council Deliberation）。但这里有一个根本性的问题：

> **LLM 是自回归模型，它不会真的"先想好再写"。** 它的"思考"和"输出"是同一个过程——每个 token 的生成都基于前面已生成的 token。

所谓的"silent pre-computation"实际上有两种可能的执行方式：

1. **LLM 在内部表征中隐式完成**——但我们无法验证它是否真的做了，也无法验证做的质量
2. **LLM 跳过这一步，直接开始生成输出**——因为指令说"silently"，所以即使跳过了，用户也看不出来

**Self-Check 中第2条**试图弥补这个问题：
> "Did the Internal Pre-computation actually shape the questions, or were they generic?"

但这同样是一个"幻觉检查"——LLM 无法回溯性地验证自己的生成过程是否被某个"预计算"影响了。

**建议：两个方向选一个——**

**方向A：让预计算可见（推荐）**
```markdown
在输出的最开头，用一个折叠块显示预计算结果：
<details>
<summary>📋 经文分析摘要</summary>
- 体裁：历史叙事
- 核心张力：[一句话]
- 误读风险：[标记]
</details>
```
这样做的好处：(1) 强迫 LLM 真的执行预计算，因为它必须输出结果；(2) 用户/带领者可以看到分析框架，增加信任；(3) 预计算的输出会进入上下文，后续生成会真正受其影响。

**方向B：取消预计算，改为嵌入式约束**
把预计算的意图直接嵌入到 DUCA 各阶段的指令中。例如，不说"先做 Genre Scan，然后根据体裁选择问题风格"，而是直接说"D 阶段的问题风格必须匹配经文体裁（叙事用场景重建，书信用逻辑流程图……）"。这样 LLM 在生成 D 的问题时自然会考虑体裁，不需要一个独立的"预计算步骤"。

### 🔴 发现3：Anti-patterns 的"负面示范效应"

**Instruction Following 专家：**

v8.0.0 中大量使用了 Anti-patterns（"never do these"），并且给出了**具体的坏例子**：

> 第182行：Bad: "Have you experienced God's grace at work? Please share."
> 第117行：Questions so simple they feel patronizing ("Who healed the paralytic?")

这里有一个被广泛记录的 LLM 行为模式：

> **当你在 prompt 中给出一个"不要做"的具体例子时，LLM 的注意力会被这个例子吸引，反而增加生成类似内容的概率。** 这被称为"负面启动效应"（negative priming paradox）。

类比：如果我说"不要想一只粉色的大象"，你脑子里立刻出现了粉色大象。LLM 的 attention 机制有类似的特性。

**建议：**
- **保留 Anti-pattern 的规则描述**（"不要生成可以用'多祷告'回答的问题"）
- **删除具体的坏例子**（"Have you experienced God's grace at work?"）
- **用好例子替代坏例子**——研究表明，给 LLM 看"好的长什么样"比告诉它"坏的长什么样"有效 3-5 倍

当前 v8.0.0 中好例子与坏例子的比例：

```
好例子（Quality Benchmark - Good）：1个
坏例子（Anti-patterns + Bad examples）：~8个
比例 = 1:8  ← 严重失衡
```

**应该反转为至少 3:1（好:坏）。**

### 🟡 发现4：多语言 Token 开销的隐性成本

**Tokenizer 专家：**

SKILL.md 是全英文编写的（~3,500 tokens），但输出要求是中文。这意味着：

1. **Prompt 是英文** → LLM 用英文"理解"指令
2. **输出是中文** → LLM 需要在生成时做"语言切换"
3. **经文引用** → 需要从英文指令语境切换到中文和合本

这个语言切换本身不是大问题（现代 LLM 处理得很好），但有一个微妙的影响：

> **英文 prompt 中的细微语义区分，在中文输出中可能被"压平"。**

例如，v8.0.0 区分了 "Answer Direction"（答题方向）和 "Common Pitfalls"（常见误读）和 "Deeper Exploration"（深入探索）——这三个英文术语的语义边界很清晰。但当 LLM 用中文输出时，"答题方向"和"深入探索"的边界容易模糊，导致内容重叠。

**建议：** 对于关键的结构性术语，在 SKILL.md 中同时给出中英文：

```markdown
#### Answer Direction / 答题方向
（核心解经洞察，不是标准答案，而是思考的轨迹）

#### Common Pitfalls / 常见误读
（人们容易犯的错，以及纠正方向）
```

这样 LLM 在生成中文输出时，对每个板块的"职责边界"更清晰。

### 🟡 发现5：Self-Check 的执行时机问题

**Attention 专家：**

Self-Check 放在 SKILL.md 的**倒数第二个板块**（第260-274行）。由于 recency effect，LLM 在生成结束时确实会"注意到"这些检查项。但问题是：

> **自回归模型的 Self-Check 只能影响"还没生成的部分"。** 当 LLM 生成到最后、"看到" Self-Check 时，前面 90% 的内容已经生成完毕，无法回溯修改。

实际效果是：Self-Check 主要影响**输出的最后 10%**（通常是时间分配表和结语），对前面的 DUCA 问题和 Study Reference 几乎没有约束力。

**这就是为什么我之前建议把核心红线前置到文档开头**——利用 primacy effect 让这些约束在生成的**全过程**中都保持活跃。

**补充建议：** 除了前置核心红线，还可以在 DUCA 每个阶段的指令末尾加一句**内联检查**：

```markdown
#### A — Application
...（现有内容）...
> ⚠️ 生成后立即检查：场景是否2-4句？是否有具体利益冲突？能否用"多祷告"回答？
```

这样每个阶段都有一个"就地检查点"，而不是等到最后才统一检查。

### 🟡 发现6：缺少"输出长度锚定"

**Instruction Following 专家：**

v8.0.0 对输出的**结构**有详细规范，但对**长度**几乎没有指引。唯一的长度相关指令是 Application 场景的"2-4 sentences"。

LLM 在没有长度锚定的情况下，倾向于**过度生成**——尤其是 Study Reference 部分，因为"Answer Direction"没有字数限制，LLM 会倾向于写得越来越长（这正是我们之前观察到的"后半段质量衰减"的原因之一）。

**建议：** 为每个输出组件设定**软性长度锚定**：

```markdown
Study Reference 各组件的目标长度：
- Answer Direction：150-250字（中文）/ 100-150 words（英文）
- Common Pitfalls：每条 50-80字
- Deeper Exploration：每条一句话
- Idol Exposure：100-150字（三层递进）
- Concrete Action：包含至少一句"原话"，总计100-150字
- Worst-Case Scenario：80-120字
```

这不是硬限制，但给了 LLM 一个"锚"，防止无限膨胀。

## 第二部分：圣经学习设计专家组发现

### 🔴 发现7：DUCA 模型缺少"经文诵读"环节的设计指引

**归纳式查经法专家：**

v8.0.0 的时间分配中提到了"Scripture reading slot"，但 DUCA 模型本身**没有为经文诵读提供任何设计指引**。这是一个重大遗漏。

在归纳式查经法中，**如何读经文**直接影响后续讨论的质量：
- 是一人朗读还是分角色朗读？
- 是读一遍还是读两遍（第一遍整体感知，第二遍带着问题读）？
- 是先读完再讨论，还是边读边停？

**建议：** 在 DUCA 之前加入一个简短的 **R（Reading）环节指引**：

```markdown
### 经文诵读指引
根据经文体裁建议诵读方式：
- 叙事：分角色朗读（旁白、耶稣、其他人物），增强场景感
- 书信：一人慢读，其余人标记关键连接词（所以、因为、但是）
- 诗歌：两人交替读平行句，感受结构
建议读两遍：第一遍整体感知，第二遍带着D1的问题重读。
```

### 🔴 发现8：Correlation 阶段的神学方法论缺失

**释经学专家：**

v8.0.0 的 Correlation 阶段要求"cross-references that genuinely illuminate the passage"，但没有指定**关联的神学方法论**。这导致 LLM 可能生成三种质量差异很大的关联：

| 关联类型 | 质量 | 当前是否被引导 |
|---------|------|--------------|
| **词汇关联**（同一个词出现在不同经文） | 低 | Anti-pattern 中禁止了"verse-piling"，但没有正面引导 |
| **主题关联**（同一个神学主题的不同展开） | 中 | 有提及"recurring biblical patterns" |
| **救赎历史关联**（这段经文在"创造→堕落→救赎→完成"大叙事中的位置） | **最高** | ❌ **完全没有提及** |

**建议：** 明确要求 Correlation 优先使用救赎历史关联：

```markdown
优先选择能揭示以下关联的经文：
- 这段经文如何指向或预表基督的工作
- 这段经文在"创造→堕落→救赎→成全"的大叙事中处于什么位置
- 新约如何引用或重新诠释这段旧约经文（反之亦然）
```

这不仅提升了神学深度，也是 Risk Assessment 中"反道德主义"的最有效武器——当经文被放在救赎历史的框架中，它就不容易被简化为"行为准则"。

### 🟡 发现9：Application 场景的"文化单一性"

**职场神学专家：**

v8.0.0 的 Application 示例和约束，隐含了一个**特定的职场文化假设**——大型企业、科层制、有明确的绩效评估体系（PIP、KPI、晋升通道）。

但职场基督徒的实际场景远比这丰富：
- 创业者 / 自由职业者（没有"上司"和"下属"的概念）
- 体制内 / 公务员（"灰色手段"的含义完全不同）
- 蓝领 / 服务业（"绩效"不是核心压力，"尊严"和"生存"才是）
- 远程工作者（"同席"的物理场景不存在）

当前 prompt 的 Hard Constraint 第3条要求"specific workplace roles (e.g., your direct supervisor, your subordinate, your client, HR)"——这个列表本身就暗示了一个科层制企业的假设。

**建议：** 扩展角色列表，或改为更通用的表述：

```markdown
场景必须包含具体的关系角色——可以是上下级、同事、客户、
合作伙伴、供应商、监管方，也可以是创业合伙人、
团队成员、服务对象等。关键是角色之间存在真实的利益关联。
```

### 🟡 发现10：缺少"经文难度分级"机制

**归纳式查经法专家：**

v8.0.0 对所有经文使用同一套 DUCA 规范，但不同经文的**讨论难度差异巨大**：

| 经文类型 | 讨论难度 | 当前处理 |
|---------|---------|---------|
| 叙事（如马可福音2章） | ⭐⭐ 中等 | 默认处理 |
| 保罗书信（如罗马书9章） | ⭐⭐⭐⭐⭐ 极高 | 同样的默认处理 |
| 诗篇（如诗篇23篇） | ⭐⭐ 中等 | 同样的默认处理 |
| 启示录 | ⭐⭐⭐⭐⭐ 极高 | 同样的默认处理 |

对于高难度经文，当前的 DUCA 规范可能导致：
- D 阶段的问题对普通信徒来说太难（尤其是书信的逻辑流程分析）
- U 阶段的问题需要大量神学背景知识才能讨论
- 50分钟根本不够用

**建议：** 加入难度自适应机制：

```markdown
### 难度自适应
根据经文的讨论难度自动调整：
- 高难度经文（密集神学论证、争议性段落）：
  减少问题数量（D:2, U:1, C:1, A:1），增加每题深度
  Study Reference 中增加更多背景铺垫
- 标准难度经文：使用默认配置
在输出开头标注难度等级，帮助带领者预判准备量。
```

### 🟡 发现11：Track B（即时辅导）的规范过于单薄

**小组动力学专家 + Instruction Following 专家（联合发现）：**

Track A 占了 SKILL.md 约 **85%** 的篇幅，Track B 只有 **15行**（第234-248行）。但从实际使用频率来看，Track B（"老板让我做违背良心的事怎么办"这类即时问题）可能比 Track A 更高频。

Track B 当前的问题：
1. **没有对话深度管理**——用户第一次提问和第三次追问，应该有不同的回应策略
2. **没有"追问引导"**——好的辅导不是一次性给答案，而是通过追问帮助对方自己发现答案
3. **没有边界案例处理**——比如用户描述的情况涉及违法行为（不只是"灰色地带"），应该有不同的回应框架

**建议：** 扩展 Track B，至少加入：

```markdown
## Track B 补充规范

### 对话深度管理
- 首次回应：Core Insight + Council Perspectives + Actionable Takeaway（当前结构）
- 用户追问时：不要重复框架，而是针对追问的具体点深入
- 第三轮及以后：转向苏格拉底式追问，帮助用户自己得出结论

### 追问引导
每次回应末尾，附一个引导性问题，帮助用户深入思考：
"你觉得在这个情况下，你最害怕失去的是什么？"

### 边界升级
当用户描述的情况涉及以下内容时，升级回应框架：
- 明确违法行为 → 明确建议咨询法律专业人士，不要只用信仰框架处理
- 职场霸凌/性骚扰 → 建议保留证据、寻求HR或外部支持
- 严重伦理困境（如医疗、金融合规）→ 建议寻求该领域的基督徒专业人士意见
```

### 🟢 发现12：缺少"复用性设计"——跨经文的模式积累

**职场神学专家：**

当前 SKILL.md 把每次查经当作独立事件处理。但在真实的小组场景中，**跨经文的主题积累**是非常有价值的。例如：

- 第一周学马可福音2章（耶稣的权柄 vs 宗教体制）
- 第二周学以弗所书6:5-9（职场中的顺服与尊严）
- 第三周学但以理书3章（当信仰与体制正面冲突时）

这三周的 Application 场景可以形成一个**递进序列**——从"日常的恩典实践"到"极端情况下的信仰立场"。

**建议：** 加入一个可选的"系列模式"：

```markdown
### 系列模式（可选）
当用户指定这是系列学习的第N次时，在输出末尾附：
- 与前几次学习的主题关联（一句话）
- 下次学习的建议经文和预期主题递进方向
```

这个功能在当前版本中已经有雏形（我在输出中加了"下次学习预告"），但 SKILL.md 中没有正式规范。

## 第三部分：两个专家组的交叉发现

### 🔴 发现13：最关键的遗漏——没有"输出质量的可测量标准"

**Evaluation 专家 × 归纳式查经法专家（联合发现）：**

这是整个 SKILL.md 最大的系统性缺陷：

> **v8.0.0 有大量的"质量描述"（好的Application长什么样），但没有任何"可测量的质量标准"。**

这意味着：
1. 你无法自动化地评估输出质量
2. 你无法在不同版本之间做 A/B 测试
3. 你无法建立一个 regression test suite 来确保新版本不退化

**建议：** 建立一个**评分量表（Rubric）**，不放在 SKILL.md 中（不占 prompt 空间），而是作为一个独立的评估工具：

```markdown
# 查经方案质量评分表（独立文件，不注入prompt）

## Application 问题评分（每题0-5分）
- 0分：没有职场场景，纯信仰反思
- 1分：有职场场景但没有利益冲突
- 2分：有利益冲突但只有一个"显然正确"的选项
- 3分：有真实的两难，但场景不够具体（没有角色）
- 4分：有真实两难+具体角色，但与经文的连接较弱
- 5分：有真实两难+具体角色+明确要求从经文出发回应

## Study Reference 评分（每题0-5分）
- 0分：没有提供
- 1分：泛泛的属灵教导，与问题脱节
- 2分：回应了问题但只有单一视角
- 3分：多视角但没有张力
- 4分：多视角+有张力，但缺少具体行动
- 5分：多视角+张力+具体行动+最坏情况+多层偶像曝光
```

这个 Rubric 可以用来：
- 人工评估每次输出的质量
- 作为 prompt 迭代的基准（v8 vs v9 的输出，用同一个 Rubric 打分）
- 未来甚至可以让另一个 LLM 用这个 Rubric 自动评分

**这些原则我会记住，应用到未来所有 skill 的设计中。** 核心原则总结：

1. **注意力经济：** 最重要的指令放在 prompt 的开头和结尾
2. **正面示范 > 负面禁止：** 好例子比坏例子有效 3-5 倍
3. **可执行性：** 不给 LLM 无法真正执行的"幻觉指令"
4. **长度锚定：** 没有长度指引 = 过度生成
5. **内联检查 > 末尾检查：** 检查点要分布在生成过程中
6. **可测量性：** 没有 Rubric 的质量标准等于没有标准
